# dsb-bench

Standalone benchmark harness for [DeathStarBench](https://github.com/delimitrou/DeathStarBench).

Apps are deployed via `docker compose`; load is generated by the DeathStarBench
**wrk2 fork** (`DeathStarBench/wrk2/`), which has built-in rate control (`-R`)
and inter-arrival distribution selection (`-D fixed|exp|zipf|norm`). Three apps
are supported, selected via the `APP` env var:

| `APP` | Frontend | Workloads (`WORKLOAD`) |
|---|---|---|
| `socialNetwork` (default) | `http://localhost:8080` | `compose-post` (default), `read-home-timeline`, `read-user-timeline`, `mixed-workload` |
| `mediaMicroservices` | `http://localhost:8080` | `compose-review` (default) |
| `hotelReservation` | `http://localhost:5000` | `mixed-workload_type_1` (default) |

## Prerequisites

```bash
./setup.sh
```

Detects your distro and installs Docker + the compose plugin, build tools
(`gcc`, `make`, `openssl`, `zlib`, `luajit`), `git`, `curl`, `python3`, and the
Python packages `numpy matplotlib aiohttp`. You may need to re-login afterwards
so your user picks up docker-group membership.

> `install.sh` clones DeathStarBench into `dist/DeathStarBench` and builds
> everything there — nothing system-wide. `DSB_DIR` (default
> `dist/DeathStarBench`) points at that checkout; override it to reuse an
> existing clone. `DSB_REPO` / `DSB_REF` select the git URL and branch/tag.

## Quick Start

```bash
./install.sh                     # build wrk2, warm app images (one-time)
./run.sh                         # single run -> rst/ (socialNetwork / compose-post)
APP=hotelReservation ./run.sh    # benchmark a different app
./run.sh --plot                  # also render a latency-percentile PNG
./batch-run.sh                   # sweep many configs (edit arrays at top first)
```

## Scripts

### `install.sh`

Clones DeathStarBench into `dist/DeathStarBench` (with the wrk2 LuaJIT
submodule), builds wrk2, and warms the docker images. Re-running updates the
existing clone to `DSB_REF`. `APP=all` (default) warms every app; `APP=<name>`
just one. `PULL=0` skips the image step. `DSB_REPO` / `DSB_REF` override the
source git URL and ref.

### `start.sh` / `stop.sh`

Bring an app up / down. `start.sh` runs `docker compose up -d`, waits for the
frontend, and performs one-time data initialization (guarded by a marker in
`.state/`). `stop.sh` runs `docker compose down -v` (clean slate) and clears the
marker so the next start re-initializes consistently.

| Env var | Script | Description |
|---|---|---|
| `APP` | both | Which app (default `socialNetwork`) |
| `REINIT` | start | Force re-running data init even if the marker exists |
| `WAIT_TIMEOUT` | start | Seconds to wait for the frontend (default 180) |
| `KEEP_DATA` | stop | Keep volumes + marker (faster restart, no `-v`) |

Data initialization per app:
- **socialNetwork**: `scripts/init_social_graph.py --graph=socfb-Reed98 --compose`
- **mediaMicroservices**: `write_movie_info.py` + `register_users.sh` + `register_movies.sh`
- **hotelReservation**: none (data baked into the images)

### `bench.sh`

Runs one wrk2 benchmark. App + workload come from `APP` / `WORKLOAD`; the rest
from options. By default it starts the app before and stops it after; pass
`--keep-server` to reuse a running one.

| Option | Default | Description |
|---|---|---|
| `-t THREADS` | WRK_CORES count, else `nproc` | wrk2 threads |
| `-c CONNECTIONS` | `100` | Concurrent connections |
| `-d DURATION` | `30` | Duration (seconds; `30s`, `2m` also accepted) |
| `--rps RPS` | `1000` | Target req/s (wrk2 `-R`; **required**, > 0) |
| `--rps-dist NAME` | `fixed` | Inter-arrival distribution (wrk2 `-D`): `fixed`, `exp`, `zipf`, `norm` |
| `--timeout SEC` | — | Socket timeout |
| `--no-latency` | — | Omit `-L` (skip the latency spectrum) |
| `--keep-server` | — | Don't start/stop the app |

`WRK_CORES` pins wrk2 to specific CPU cores (e.g. `0-3` or `0,2,4,6`); the core
count must equal `-t THREADS` (same rule as nginx-bench).

#### `--rps-dist` distributions (native wrk2 `-D`)

| Name | Behavior |
|---|---|
| `fixed` | constant request interval (default) |
| `exp` | exponential interval; same mean as `fixed` (Poisson arrivals) |
| `zipf` | zipf-distributed interval; same mean as `fixed` |
| `norm` | send next request immediately after the last |

### `run.sh`

Single-run driver: ensures the app is up + initialized (idempotent `start.sh`),
runs one benchmark via `bench.sh --keep-server`, and saves the full output (with
UTC `Start:` / `End:` stamps) to `rst/`. Filenames encode every parameter plus a
UTC timestamp, so runs accumulate without overwriting:

```
rst/dsb-socialNetwork-compose-post_wrk-cpu20-29-t10-c300-d30-R1000-Dfixed_20260609-143005.out
```

**Override env vars:** `APP`, `WORKLOAD`, `WRK_CORES`, `THREADS`, `CONNECTIONS`,
`DURATION`, `TARGETRPS`, `TARGETRPS_DIST`, plus optional `RUN_TAG`. If `WRK_CORES`
is set but `THREADS` isn't, `THREADS` is derived from the core count.

`--plot` renders a `.png` latency-percentile curve via `plot-latency.py`. The
`.out` file is self-documenting — it ends with the exact `Plot:` command to
render later.

#### `plot-latency.py`

Parses the wrk2 `-L` "Detailed Percentile spectrum" from an `.out` file and
plots latency (ms) against percentile on a logarithmic `1/(1-p)` x-axis — the
canonical HdrHistogram tail view — with p50/p90/p99/p99.9 markers.

```bash
python3 ./plot-latency.py rst/<run>.out rst/<run>.png
```

### `batch-run.sh`

Sweeps the Cartesian product of setting arrays, optionally repeated `REPEATS`
times. Each iteration calls `run.sh`; the repeat number is tagged into the
filename. When the `APP` changes between iterations the previous app is torn
down first (social/media share port 8080). Edit the arrays at the top:

```bash
REPEATS=3
APP_arr=(socialNetwork hotelReservation)
CONNECTIONS_arr=(100 300 500)
TARGETRPS_arr=(1000 5000 10000)
TARGETRPS_DIST_arr=(fixed exp)
# ...
```

## Examples

```bash
# Default social-network compose-post run, plotted
./run.sh --plot

# Read-heavy workload at a fixed rate
APP=socialNetwork WORKLOAD=read-home-timeline TARGETRPS=5000 ./run.sh

# Poisson arrivals at the same target rate
TARGETRPS_DIST=exp ./run.sh

# Hotel reservation mixed workload, longer duration
APP=hotelReservation DURATION=120 CONNECTIONS=200 ./run.sh

# Pin wrk2 to its own cores (leave the rest for the app containers)
WRK_CORES=20-29 ./run.sh

# Direct bench.sh against an already-running app
APP=hotelReservation ./start.sh
APP=hotelReservation ./bench.sh -t 8 -c 200 -d 60 --rps 2000 --keep-server
```

## Directory Layout

```
dsb-bench/
├── setup.sh          install docker + compose + build/python deps
├── install.sh        build wrk2, warm app images
├── start.sh          docker compose up + data init
├── stop.sh           docker compose down -v + clear marker
├── bench.sh          run wrk2 (-t -c -d -R -D -L -s <lua> <url>)
├── run.sh            single run -> rst/, optional --plot
├── batch-run.sh      Cartesian sweep
├── plot-latency.py   latency-percentile PNG from wrk2 -L output
├── config/
│   └── apps.sh       per-app compose file, frontend URL, workloads, init steps
├── dist/             cloned DeathStarBench + built wrk2 (created by install.sh)
│   └── DeathStarBench/
├── rst/              benchmark outputs (.out, .png)
└── .state/           per-app "initialized" markers
```
